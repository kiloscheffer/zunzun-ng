# Active gotchas

Operational rules-of-thumb — situational notes important enough to keep grep-able, grouped by area of the codebase. Read the matching section before touching that area. Each bullet captures one rule, one tripping-hazard, or one footgun that has burned someone (or would burn the next person).

## Environment and venv

- `.venv/` must be excluded from cloud-sync clients (Dropbox, OneDrive, iCloud). uv's default hardlink mode on Windows shares inodes between `.venv/` and the global uv cache (`%LOCALAPPDATA%\uv\cache\`); a sync client watching `.venv/` doubles the storage and breaks the hardlink relationship on cross-machine sync, silently corrupting Python imports (manifests as OS error 396 or hard-to-trace `ImportError`s). With the exclusion in place, no special handling is needed (verified 2026-04-28). Workaround if the exclusion isn't possible: prefix `uv` commands with `UV_LINK_MODE=copy`.
- Cold-cache smoke flakiness. The first smoke run after `rm -rf .venv && uv sync` (especially after a pyeq3 reinstall) can time out on 3D scenarios because spawn workers compile `.pyc` files on first import. Re-running on the warm venv passes.
- `rm -rf .venv` may fail with "Device or resource busy" on Windows when transient background processes (Dropbox indexers, Windows Search, etc.) hold handles open momentarily. PowerShell's `Remove-Item .venv -Recurse -Force` uses native Win32 calls and handles these gracefully where bash's `rm -rf` (via MSYS POSIX-emulation) does not.
- Avoid running pytest + smoke in parallel right after a `uv lock` that changed any source URL — they'll race for cache locks.

## Spawn LRP pattern

- Never call `os.getloadavg`, `/proc`, `vmstat`, or `os.popen` directly from view or LRP code. Extend `zunzun/platform_compat.py` instead — it shims the platform-specific calls (Linux uses real loadavg; Windows synthesises one from psutil).
- `os.fork()` and `os._exit()` no longer appear in the codebase. Adding them will break Windows compatibility; prefer `multiprocessing.Process(spawn)` and plain `return` respectively. The `fork-pattern-reviewer` subagent in `.claude/agents/` audits for accidental regressions.
- Per-fit worker count is resolved by `zunzun.parallel_pool.resolve_max_workers()`: explicit arg > `ZUNZUN_MAX_WORKERS` env > `settings.MAX_PARALLEL_WORKERS` > auto-detect `min(cpu_count, available_RAM_KiB / 200_000)`. Result is always clamped to the hardware ceiling. Modern Python 3.14 + numpy 2.4 + scipy 1.17 workers use ~140 MB RSS each (down from the original 750 MB pessimistic estimate that drove an old 4-worker hard cap, removed 2026-05-28). On a low-RAM box, set `ZUNZUN_MAX_WORKERS=4` as an emergency throttle. `platform_compat.get_parallel_process_count()` delegates here and additionally throttles down under high system load (load1 > cpu_count + 0.5/1.0/1.5 → reduces to 3/2/1).
- `FitPool` lifetime is per-LRP-child, not per-Django-request. Create instances inside `PerformAllWork()` (which runs in the spawned LRP child), NOT in `__init__()` (which runs in the Django request handler before spawn — the parent's `multiprocessing.Process(spawn)` snapshot doesn't pickle a live `ProcessPoolExecutor`). Tear down in `PerformAllWork()`'s `finally` block. `CheckIfStillUsed()` tears down — via the shared `_teardown_abandoned_fit()` helper, which calls `self.fit_pool.shutdown(wait=False, cancel_futures=True)` and `terminate()`s live workers — on ANY of three abandonment signals: a foreign `process_id` in the row, a stale (>300 s) heartbeat, OR a **missing row** (`get_status("process_id") is None`). The missing-row case means a newer dispatch superseded this one and deleted its row (delete-prior-row in `LongRunningProcessView`); it MUST tear down rather than early-return, because once the row is gone the superseded child can no longer observe the other two signals, so a CPU-heavy fit would otherwise keep saturating cores until it finishes on its own.
- `FitPool.__init__` injects `OMP_NUM_THREADS / OPENBLAS_NUM_THREADS / MKL_NUM_THREADS=1` via `os.environ` (using "treat empty string as unset" semantics, not `setdefault`) before creating the executor. Spawn workers inherit these and use single-threaded BLAS — parallelism comes from the worker count, not from BLAS threads. Without this, 22 concurrent workers × 22 BLAS threads each → OpenBLAS init memory bomb on big-core Windows boxes (`OpenBLAS error: Memory allocation still failed after 10 retries` → `BrokenProcessPool`). Power users wanting BLAS threading inside one worker can set `OMP_NUM_THREADS=8` in their shell explicitly; FitPool only fills in empty/unset values.
- Inner parallel phases (`FunctionFinder.PerformWorkInParallel`, `CreateOutputReportsInParallelUsingProcessPool`, `StatisticalDistributions.PerformWorkInParallel`) all submit to the same `self.fit_pool` (inherited from `StatusMonitoredLongRunningProcessPage`). Worker reuse across phases amortizes the ~2 s spawn-import cost over the entire fit. On `BrokenProcessPool`, raise `_ReportsPipelineAborted` (sentinel exception in `StatusMonitoredLongRunningProcessPage`) — caught by `PerformAllWork`'s try/finally so the subsequent PDF/HTML stages are skipped and the user-visible error status survives, rather than the pipeline silently continuing to a broken empty results page.

## Sessions and state

- Every session write goes through `save_with_retry(session)` from `zunzun/session_helpers.py`; every session read goes through `load_with_retry(session, key, default=None)` from the same module. Both wrap the 100-retry @ 10Hz loop that handles SQLite contention from concurrent spawn children. Direct `session.save()` or `session[key]` calls in spawn-child code are a bug. Spawn children open fresh DB connections (vs. fork's inherited ones), so lock contention is arguably more relevant post-migration, not less.
- Session values are stored as JSON via `NumpySessionSerializer` (wired in `settings.SESSION_SERIALIZER`, defined in `zunzun/session_helpers.py`). numpy scalars and arrays produced by pyeq3 are coerced to plain Python primitives automatically at `session.save()` time — callers do NOT need to cast manually (the old `_json_native` per-call-site helper was removed once the serializer landed). Anything else stored must still be JSON-encodable (no sets, datetimes, live scipy objects, etc.).
- **Status store is a per-dispatch ORM row, not a session blob.** `LRPStatus` (`zunzun/models.py`, table `zunzun_lrpstatus`) holds `current_status`, `start_time`, `last_status_check`, `redirect_to_results`, `parallel_count`, `process_id`, and `state`; the autopk `id` is the dispatch identity. The current dispatch's row pk is stored in `request.session["lrp_status_pk"]`. Status writes go through `update_status(**fields)` on `StatusMonitoredLongRunningProcessPage` — an unconditional single-row `LRPStatus.objects.filter(pk=...).update(...)`. Reads go through `get_status(field, default=None)`. There is no ownership check: each fit child writes only its own row. The old apparatus (`_we_own_status_slot`, `_publish_terminal_error`, `dispatched_at`) has been deleted. `migrate` now creates the `zunzun_lrpstatus` table in addition to `django_session` — no extra command, but operators should know this app table exists.
- **`LRPStatus` lifecycle is an explicit `state` field, not derived fields.** `LRPStatus.State` is a `TextChoices` enum with three values: `INITIALIZING` (default, row just created), `RUNNING` (child claimed the row), and `TERMINAL` (fit finished — success, abort, or error). Lifecycle transitions go through two classmethods on `LRPStatus`: `mark_running(pk, pid)` (INITIALIZING → RUNNING, sets `process_id`); `mark_terminal(pk, *, redirect=None, current_status=None, parallel_count=None)` (→ TERMINAL, always sets `state=TERMINAL` + `process_id=0` atomically so the terminal tuple can never be set partially; optional fields only written when passed so a bare `mark_terminal(pk)` never clobbers a redirect a prior stage already published). The per-user gate in `LongRunningProcessView` reads `state` directly: `state == RUNNING` + `process_id > 0` + heartbeat within 300s = active; `state == INITIALIZING` + `start_time` within 60s = pending. `StatusView` and `StatusUpdateView` key terminal detection on `state == TERMINAL`. Do NOT key terminal detection off `redirect_to_results` alone — `StatusView` clears it on serve, so a mid-fit crash or a served success would both look like "not yet done" if you checked only the redirect. The durable terminal signal is `state == TERMINAL`.
- **`PerformAllWork`'s `finally` stays non-terminal by design.** The `finally` block in `StatusMonitoredLongRunningProcessPage.PerformAllWork` only calls `self.update_status(process_id=0)` — it deliberately does NOT set `state=TERMINAL`. On the unhandled-exception path the exception propagates to `_run_fit_child`'s `except` handler, which writes a terminal error artifact and calls `LRPStatus.mark_terminal(...)` to publish the error redirect. If `finally` pre-set `state=TERMINAL`, that handler's guard (`current_state == TERMINAL` → don't clobber) would skip the error redirect and orphan the artifact. So `state=TERMINAL` is set only by real terminal writers: the success path at the end of the `try` block (`self.mark_terminal()`), the abort sites inside `_ReportsPipelineAborted` handlers, or `_run_fit_child`'s exception handler.
- **`_finalize_row_if_child_dead` is the reader-side backstop.** A fit child that vanished without setting `state=TERMINAL` (SIGKILL / OOM / crash, or a failed terminal write) leaves `process_id` set and `state` non-terminal — the poll loop would run forever. `StatusView` and `StatusUpdateView` both call `_finalize_row_if_child_dead(row)` which checks `pid_is_alive(row.process_id)`; if the pid is gone, it calls `LRPStatus.mark_terminal(row.pk)` so the next poll cycle ends cleanly.
- **Data and functionfinder stores are still session blobs.** Two `SessionStore`s per user — keys stored in the main request session as `session_key_data` / `session_key_functionfinder`. The helpers `save_with_retry` / `load_with_retry` handle persistence (status routing through `SaveDictionaryOfItemsToSessionStore("status", ...)` was removed when the status row landed). Because these blobs are shared per-user across dispatches (not per-dispatch like the status row), the child's terminal `RenderOutputHTMLToAFileAndSetStatusRedirect` (base + the `FunctionFinder` override) bails early when its own status row is gone (`get_status("process_id") is None`): a superseded child must not clobber the winning dispatch's `functionfinder`/`data` blobs that `/FunctionFinderResults/` and `EvaluateAtAPointView` later read.

## Templates and URLs

- `/CommonProblems/` is a case-sensitive CapitalCase URL; the on-disk directory is lowercase (`commonproblems/`). The bare trailing-slash URL `/CommonProblems/` is explicitly routed to serve `index.html` because Django's `static()` helper doesn't auto-index.
- The coefficient-picker `<td>` cells in `templates/zunzun/divs/{polyfunctional,polyrational,polynomial_customization}_selection_div.html` are JS-coupled, but via modern, explicit hooks (the legacy `document.all`/`document.layers`/`.style.backgroundColor` pathway is gone). Selection *state* lives in the `selected` CSS class (`isSelected`/`setSelected` in `JavascriptCommonToFunctionMatrices.js` are the only read/write points). Each cell's `data-flag` attribute names its paired hidden `<input>`; `setSelected` writes that input on every toggle, and the input's *initial* value is server-rendered from the `selected` bool — so the form is correct at rest (no submit-time sweep). Cell `id` (`CPX...`) is still read by the per-matrix preview builders. Do not change a cell's `id`, `data-flag`, or the `selected`-class protocol without updating the matching JS.

## Files and directories

- `static/` (committed assets) and `temp/` (runtime outputs) serve two separate URL prefixes — `STATIC_URL = '/static/'` vs `MEDIA_URL = '/temp/'`. For Python-side paths use `settings.STATIC_FILES_DIR` and `settings.TEMP_FILES_DIR` respectively. They are different paths since the 2026-04-28 static-files restructure; before that, both lived under `temp/`.
- `temp/` is auto-trimmed by `HomePageView`'s housekeeping when total size exceeds `MAX_TEMP_DIR_SIZE_IN_MBYTES` (default 500).
- LRP trace logging routes through `logging.getLogger("zunzun.LongRunningProcess.*")`. Default level is `WARNING` (silent). To enable per-step tracing in spawn children, set `ZUNZUN_LRP_LOG_LEVEL=DEBUG` — output flows into `temp/{pid}.log` because `_setup_child_root_logging()` at the top of `_run_fit_child` installs the per-pid `FileHandler` on the root logger BEFORE any `PerformAllWork` code runs. `FitPool` workers are sub-children of the LRP child and DO NOT inherit its handlers (each is its own fresh spawn); `FitPool.__init__` wraps any caller-supplied initializer in `_worker_initializer` so the same per-worker-pid FileHandler installs before any worker code runs — without this, `logging.exception` calls in the `ParallelWorker_*` / `parallelWorkFunction` / animation classes would fall back to stderr and the user-facing "see log file" message would point at a missing file. Parent-side trace messages (e.g. from `SetInitialStatusDataIntoSessionVariables`) are not routed by default — add a handler to `settings.LOGGING` or run with `runserver --verbosity` to see them. The old `pid_trace.py` shim (dormant-by-design `return`-at-top no-op) was removed when the per-LRP logger landed; bare positional `pid_trace.pid_trace()` markers were deleted (no useful message), text-bearing ones converted to `_logger.debug(...)`.

## Filename grammar in temp/

- Artifact filenames follow `zun_<pid_b36_4>_<ms_b36_8>_<anchor_3>_<rank_2>.{ext}` (per-component) or `zun_<pid_b36_4>_<ms_b36_8>_zun_00.{ext}` (page-level). Helpers live in `zunzun/LongRunningProcess/_unique.py`: `new_unique_string()`, `page_artifact_filename/path/url()`, and the `b36()` formatter.
- Anchor namespace reservations: `zun` is the page-level anchor (PDF, result HTML); `h` prefix is reserved for parametrized histogram instances (`StatisticalDistributionHistogram` uses `h` + 2-char base36 of `distributionIndex`). No other anchor may start with `zun` or `h`.
- `b36()` never truncates: values too large for the requested width produce a longer string. Rank field is sized for 0..1295 (FunctionFinder cap ~1k); larger values produce 3+ char suffixes and break fixed-width sortability for those rows specifically — accepted trade-off.

## FunkLoad legacy

- FunkLoad is not in `pyproject.toml`. Its `setup.py` uses `ez_setup`, which was removed from modern setuptools, so it can't be installed under the uv-managed Python 3.14 environment.
- Its assertion strings in `funkload_tests/test_Simple.py` are also stale under modern numpy/scipy/pyeq3. The folder is preserved as historical reference; do not invest in re-running it. Port individual assertions to pytest or to the smoke script if needed.

## Deploy

- `settings.py` ships with empty placeholders for `SECRET_KEY`, `EXCEPTION_EMAIL_ADDRESS`, `FEEDBACK_EMAIL_ADDRESS`, `EMAIL_HOST_USER`, and `EMAIL_HOST_PASSWORD`. Email sending is gated on these being truthy (see `FeedbackView`, the exception handler in `LongRunningProcessView`), so leaving them blank silently disables email rather than crashing.
- `DEBUG` is toggled automatically by looking for `'runserver'` in `sys.argv` (see `settings.py`), so running under Waitress or WSGI disables debug regardless of env vars.
- Rate limiting is always in effect (no install-time gating). To disable for local testing, set `RATELIMIT_ENABLE = False` in `settings.py`. When a caller exceeds the rate, the `@middleware.rate_limit_sleep` decorator (stacked below `@ratelimit` on each rate-limited view) applies a 5-second `time.sleep` before the view body runs — this specifically breaks polling endpoints (`StatusUpdateView` is intentionally not decorated with `@ratelimit` for this reason).
