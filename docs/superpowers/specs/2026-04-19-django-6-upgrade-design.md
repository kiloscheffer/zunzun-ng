# Django 5.2 → 6.0 + Python 3.11 → 3.14 Upgrade — Design

**Date:** 2026-04-19
**Branch (planned):** `django-6-upgrade` in worktree `../zunzunsite3-django6/`
**Status:** Brainstorm complete, awaiting user sign-off before plan.

## 1. Scope & Goals

Bump pinned Django from `>=5.2,<5.3` to `>=6.0,<6.1` (STS, end-of-life ~December 2026) and pinned Python from `>=3.11` to `>=3.14`. Motivation is "stay current" — no specific 6.0 feature is needed. Preserve every pattern established by the prior 2.2 → 5.2 migration (JSON session storage, django-ratelimit, re_path URLs, plain `render`, spawn-based LRP, 82 pytest tests, 8-scenario smoke).

**No functional or architectural changes.** Expected outcome is two pin bumps, a `uv.lock` regeneration, and a handful of doc updates.

**Deliverables:**

- `pyproject.toml` pins bumped to Django 6.0 + Python 3.14.
- `uv.lock` regenerated and committed.
- `manage.py check` clean; 82 pytest tests pass; 8-scenario smoke test passes on Windows.
- `CLAUDE.md` Django version paragraph, dependencies paragraph, and any `3.11` references updated to reflect reality.

**Non-goals:**

- No code changes unless a forced deprecation in 6.0 requires one.
- No fix for the 3D smoke deadlock documented in `TODO.md`.
- No fix for the spline/UDF session round-trip gaps documented in `TODO.md`.
- No addition of new features, tests, or LRP subclasses.
- No upstream patch to pyeq3 even if its classifiers are stale.
- No push to `origin` (user preference — same as last upgrade).

## 2. Constraints & Context

- Master is at commit `fa2e250` with Django 5.2.13 + Python 3.11 running cleanly.
- Python 3.14.4 is already installed and available via uv on the target Windows box (`uv python list` confirms).
- The user's repo lives under `C:\Dropbox\git\zunzunsite3\`; Dropbox-backed filesystem requires `UV_LINK_MODE=copy` for every `uv` invocation (previously memorized).
- pyeq3 12.6.1's PyPI classifiers list Python 3.7–3.10 only and `requires_python` is empty. The package has been running on 3.11 successfully for weeks; it is mostly a pure-Python wrapper over numpy/scipy, so the classifier lag is cosmetic in practice. Python 3.14 inherits the same "works despite stale classifier" situation, gated only on numpy/scipy/matplotlib having 3.14 wheels — which they do by April 2026.
- Django 6.0's single new hard requirement is Python ≥ 3.12. Going to 3.14 is user-choice (stay-current), not a Django requirement.

## 3. Decisions (Locked In)

| # | Decision | Alternatives considered |
|---|---|---|
| 1 | **Target Django 6.0.4 (or latest 6.0.x)** | Stay on 5.2 LTS; wait for 6.2 LTS in April 2027 |
| 2 | **Target Python 3.14.4** | Stay on 3.11; bump to 3.12 (the Django 6.0 minimum); bump to 3.13 |
| 3 | **No functional code changes unless the upgrade forces one** | Opportunistic cleanup of any pattern newly deprecated in 5.x/6.0 |
| 4 | **Two separate phase commits: Python bump first, Django bump second** | One atomic commit; staged across branches |
| 5 | **Reuse existing 82 pytest + 8-scenario smoke as the acceptance bar** | Add new 6.0-specific tests |

## 4. Components Touched

### 4.1 `pyproject.toml` (two one-line changes)

- `requires-python = ">=3.11"` → `requires-python = ">=3.14"`
- `"django>=5.2,<5.3"` → `"django>=6.0,<6.1"`

### 4.2 `uv.lock`

Regenerated by `uv sync` after each pin change. Expected drift: Django 5.2.13 → 6.0.x, asgiref / tzdata possibly updated, numpy / scipy / matplotlib / reportlab possibly bumped to whatever's 3.14-compatible.

### 4.3 `CLAUDE.md` (doc updates)

Two paragraphs to update after the pin changes land:

- **"Dependencies" paragraph** currently reads `Django (pinned \`>=5.2,<5.3\`), django-ratelimit, pyeq3, ...`. Update to `Django (pinned \`>=6.0,<6.1\`), django-ratelimit, pyeq3, ...`.
- **"Django version." paragraph** currently describes the 5.2 LTS timeline and points at the 2026-04-18 spec. Rewrite to describe 6.0 STS + next LTS (6.2, April 2027) and cross-reference both the 2026-04-18 spec and this one.

### 4.4 Grep sweep for `3.11` in docs

Repo-wide `grep -rn "3\\.11"` to find any stale Python version literal. Expected hits: README, deployment docs, maybe `docs/superpowers/specs/*` history (those stay — they're point-in-time records). Surgical edits only in docs that describe current state.

### 4.5 `settings.py` — possible `DEFAULT_AUTO_FIELD`

Django 6.0 changes the default from `AutoField` to `BigAutoField`. The project has no models (`INSTALLED_APPS` includes only `django.contrib.sessions` and `django.contrib.staticfiles` and `zunzun` itself). Most likely `manage.py check` stays clean and no edit is needed. If it warns, add `DEFAULT_AUTO_FIELD = 'django.db.models.AutoField'` explicitly to silence (or accept the new default if the warning explicitly says "no action needed").

## 5. Execution Phases

Five phases, the smallest of any migration so far.

1. **Phase 0 — Setup.** Create worktree `../zunzunsite3-django6/` with branch `django-6-upgrade` off master. Run `UV_LINK_MODE=copy uv sync` to verify the worktree installs cleanly on the current (5.2 + 3.11) stack. Run baseline: full pytest + smoke green. If any baseline fails, do NOT proceed.

2. **Phase 1 — Python bump (Django 5.2 kept).** Change `requires-python` in `pyproject.toml` to `>=3.14`. Run `UV_LINK_MODE=copy uv sync`. Run `manage.py check`, full pytest, full smoke. Commit. Rationale for keeping Django fixed during this phase: any regression isolates cleanly to the Python version.

3. **Phase 2 — Django bump.** Change `django` pin in `pyproject.toml` to `>=6.0,<6.1`. Run `UV_LINK_MODE=copy uv sync`. Run `manage.py check`, full pytest, full smoke. Commit. If any fails, diagnose and fix before commit (likely a single-line API or setting adjustment).

4. **Phase 3 — Documentation.** Update `CLAUDE.md` two paragraphs. Grep-and-surgically-fix any `3\\.11` references that describe current state. Commit.

5. **Phase 4 — Merge.** Local `git merge --no-ff django-6-upgrade` on master from the main checkout. Verify `pytest` + `smoke` one final time on master. Do NOT push to `origin` unless explicitly asked.

## 6. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Python 3.14 wheels unavailable for numpy / scipy / matplotlib. | `uv sync` fails in Phase 1 with a clear error. Fallback: drop Python pin to 3.13 or 3.12 and continue Phase 2 on Django 6.0 regardless. |
| pyeq3 has a latent 3.14 incompatibility (stale classifiers suggest upstream inactivity). | Phase 1 baseline (Python-only bump) exercises pyeq3 end-to-end via smoke. Failure isolates to Python, not Django. Options: pin Python lower, or accept a small local patch to pyeq3. |
| Django 6.0 removes a runtime API our code still uses. | Phase 2 `manage.py check` + pytest flag it. Given our code only touches `re_path`, `render`, `HttpResponse`, `HttpResponseRedirect`, `SessionStore`, `@cache_page`, `@cache_control`, `EmailMessage`, and middleware — all long-stable — this is unlikely. |
| `multiprocessing.get_context("spawn")` semantics changed between 3.11 and 3.14. | Smoke test exercises spawn end-to-end in 7 of 8 scenarios. Any behavior change surfaces there. |
| `DEFAULT_AUTO_FIELD` change triggers `check` warning. | Explicit `DEFAULT_AUTO_FIELD = 'django.db.models.AutoField'` added to `settings.py` if warning appears. No proactive setting. |
| `session_db/db.sqlite3` incompatible after Django 6.0 bump. | Session DB is gitignored and throwaway. Recipe: `rm session_db/db.sqlite3 && uv run python manage.py migrate`. Same pattern used for the prior worktree. |
| Dropbox `UV_LINK_MODE=copy` requirement missed by subagent. | Auto-applied via memorized project note (`project_dropbox_uv_linkmode.md`). |
| `uv python install 3.14.4` needed on target box. | Already installed (`uv python list` confirms). No action. |

## 7. Acceptance Criteria

- `uv run python -c "import django; print(django.get_version())"` → output starts with `6.0.`
- `uv run python -c "import sys; print(sys.version_info[:3])"` → `(3, 14, N)` where N ≥ 4.
- `uv run python manage.py check` → `System check identified no issues (0 silenced).`
- `uv run pytest tests/ -v` → 82 tests pass, 0 failures.
- `uv run python scripts/smoke_test.py` → `SMOKE OK: all scenarios passed` with 8 scenarios.
- `pyproject.toml` shows `requires-python = ">=3.14"` and `"django>=6.0,<6.1"`.
- `CLAUDE.md` no longer describes Django 5.2 as current, and no `3.11` string remains in any current-state doc.
- `grep -rn "django>=5\\.2,<5\\.3\\|5\\.2 LTS\\|Python 3\\.11" .` returns only matches inside `docs/superpowers/specs/` (historical specs) and `uv.lock`.
